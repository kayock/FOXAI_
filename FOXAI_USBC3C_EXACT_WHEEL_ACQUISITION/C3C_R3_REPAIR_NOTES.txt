REASON FOR R3
The R2 rerun resumed correctly and accepted 74 exact wheels before stopping
fail-closed while validating:

  setuptools-81.0.0-py3-none-any.whl

R2 searched the entire archive for every path ending in .dist-info/RECORD,
.dist-info/WHEEL, and .dist-info/METADATA, then required the archive-wide count
of each name to be exactly one. Setuptools legitimately carries vendored package
metadata deeper inside its own package tree. Those nested vendored .dist-info
files are payload files covered by the primary wheel RECORD; they are not extra
primary metadata directories for the setuptools wheel.

EXACT REPAIR
R3 identifies the wheel's primary metadata only when RECORD, WHEEL, and METADATA
are direct children of a top-level *.dist-info directory. It still requires:
1. exactly one top-level RECORD;
2. exactly one top-level WHEEL;
3. exactly one top-level METADATA;
4. all three files to share the same top-level dist-info directory;
5. METADATA project and version to match the reviewed manifest;
6. WHEEL tags to match the reviewed filename and portable Python tags;
7. every actual archive file except the primary RECORD itself to have an exact
   size and cryptographic digest in the primary RECORD.

Nested vendored .dist-info files are treated as ordinary payload files. They do
not count as additional primary metadata, but they remain subject to path safety,
duplicate, encryption, symlink, size, and digest verification through the
primary RECORD.

NEGATIVE TESTS RETAINED
- A wheel containing multiple top-level dist-info metadata directories fails.
- A real archive file omitted from the primary RECORD fails.
- Unsafe paths, duplicate entries, encrypted entries, symlinks, identity
  mismatches, tag mismatches, outer size mismatches, and outer SHA-256 mismatches
  continue to fail closed.

No acquisition manifest, approved URL, package version, hash, size, source host,
install boundary, runtime boundary, or launcher boundary changed.
